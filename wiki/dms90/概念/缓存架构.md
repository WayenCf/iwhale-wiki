# 缓存架构

> DMS 采用两级缓存架构：Guava 本地缓存作为一级缓存，Redis 作为分布式刷新触发器，实现集群缓存一致性。

## 架构图

```
应用节点 A                应用节点 B
    |                        |
    v                        v
Guava Cache          Guava Cache
(本地内存)             (本地内存)
    ^                        ^
    |                        |
    +--------+---------------+
             |
          Redis
    (刷新触发器)
```

## LocalCache 实现（dms-common/cache 包）

### 缓存构建参数

| 参数 | 说明 |
|------|------|
| initialCapacity | 初始容量 |
| maximumSize | 最大容量 |
| expireAfterAccess | 访问后过期时间 |
| concurrencyLevel | 并发级别 |

### CacheKey<T> — 复合缓存键

| 字段 | 类型 | 含义 |
|------|------|------|
| tenant | String | 租户 |
| type | Class<T> | 缓存数据类型 |
| version | String | 版本 |
| key | String | 业务键 |

重写 equals/hashCode 基于 4 字段，toString() 输出 JSON。

## Redis 分布式刷新触发器

### RedisCacheRefreshTrigger

- **条件**：`ftf.dms.cache.refresh.enabled=true`
- **实现**：CommandLineRunner，启动后每秒轮询 Redis Hash
- **Redis Hash 结构**：
  ```
  refreshCache → {
    expectedRefreshTime: 1713600000000,
    node1_lastRefreshTime: 1713600000000,
    node1_lastCheckTime: 1713600000100,
    node2_lastRefreshTime: 1713600000000,
    node2_lastCheckTime: 1713600000100
  }
  ```
- **刷新逻辑**：当 `expectedRefreshTime > lastRefreshTime` 时，执行 LocalCache.clear()
- **节点清理**：超过 6 秒未更新 lastCheckTime 的节点视为已停止，清除其 key
- **nodeId**：取自环境变量 CLOUD_NODE_ID 或主机名

### CacheRefreshController

- **端点**：GET `/cache/refresh`
- **逻辑**：刷新本地缓存，集群模式下通过 Redis 协调各节点刷新，超时 4s 抛异常

## 业务层使用场景

### ReleaseServiceImpl — 双重检查锁缓存

```java
synchronized(lock) {
    Release release = LocalCache.get(cacheKey);
    if (release == null) {
        release = releaseDao.retrieve(id);
        LocalCache.put(cacheKey, release);
    }
}
```

### UserServiceImpl — 用户缓存

- `retrieve(id)` 使用 LocalCache 缓存用户对象
- 减少数据库查询

### DocEncryptor — KeyStore 缓存

- `getKeyStore(cak)` 使用 LocalCache 缓存已加载的 PKCS12 KeyStore
- 避免重复读取证书文件

### DocMakeTaskFactory — 切割规则缓存

- 切割规则（JSON → Rule → SegmentDef）缓存到 LocalCache
- 减少重复解析

### BatchDocMakeTask — 模板缓存

- `TEMPLATE_CACHE` 缓存已编译的 JasperReports 模板
- 避免重复编译

## 缓存一致性问题

### 场景

1. 节点 A 更新数据，清除本地缓存
2. 节点 B 的本地缓存仍为旧数据
3. 用户请求路由到节点 B，读到脏数据

### 解决方案

1. **触发式刷新**：数据更新时，写入 Redis `expectedRefreshTime = currentTime`
2. **被动轮询**：各节点每秒检查 Redis，发现需要刷新则清空本地缓存
3. **节点清理**：超过 6 秒未活动的节点自动从 Redis 中移除

## 与 Redis Session 的区别

| 维度 | Redis Session | Redis Cache Refresh |
|------|--------------|---------------------|
| 用途 | 会话共享 | 缓存刷新触发 |
| 数据 | 用户会话数据 | 刷新时间戳 |
| 读取频率 | 每次请求 | 每秒轮询 |
| 写入频率 | 登录/登出 | 数据更新时 |
